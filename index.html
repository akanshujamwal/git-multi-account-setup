How multi-account SSH works

GitHub identifies you by your SSH key, not by username and password. With two accounts on one machine you need two separate keys and an SSH config that maps a named alias to each key. You stop using github.com directly — every URL uses an alias instead.

1

Generate SSH key pairs

Run these commands in your terminal. Each account needs its own key file with a unique name. The -C flag adds your email as a label so you can identify keys later.

💡Already have keys? Check with ls ~/.ssh/ — if both files exist, skip to Step 2.
Generate personal key personal
# Replace the email with your personal GitHub email
ssh-keygen -t ed25519 -C "personal@email.com" -f ~/.ssh/id_ed25519_personal

# Press Enter twice to skip passphrase (or set one for security)
Generate work key work
# Replace the email with your work GitHub email
ssh-keygen -t ed25519 -C "you@company.com" -f ~/.ssh/id_ed25519_work

# Press Enter twice to skip passphrase (or set one for security)
Add both keys to SSH agent
# Start the agent
eval "$(ssh-agent -s)"

# Add both keys
ssh-add ~/.ssh/id_ed25519_personal
ssh-add ~/.ssh/id_ed25519_work

# Confirm both are loaded
ssh-add -l
2

Add public keys to GitHub

Copy each public key and paste it into the matching GitHub account. Log into the correct account before adding each key.

Copy personal public key personal
# macOS — copies straight to clipboard
pbcopy < ~/.ssh/id_ed25519_personal.pub

# Linux — print then manually copy the output
cat ~/.ssh/id_ed25519_personal.pub
Copy work public key work
# macOS — copies straight to clipboard
pbcopy < ~/.ssh/id_ed25519_work.pub

# Linux — print then manually copy the output
cat ~/.ssh/id_ed25519_work.pub
🔗Go to GitHub → Settings → SSH and GPG keys → New SSH key. Paste the key. Use a clear title like MacBook-Personal or MacBook-Work so you can identify it later.
3

Create the SSH config file

This is the most important step. The config maps each alias to the right private key so SSH automatically uses the correct identity — no manual switching needed.

📄 ~/.ssh/config
~/.ssh/config 
# ── Personal GitHub ────────────────────────────────────
Host github-personal
  HostName       github.com
  User           git
  IdentityFile   ~/.ssh/id_ed25519_personal
  IdentitiesOnly yes    # forces only this key — do not remove

# ── Work GitHub ────────────────────────────────────────
Host github-work
  HostName       github.com
  User           git
  IdentityFile   ~/.ssh/id_ed25519_work
  IdentitiesOnly yes    # forces only this key — do not remove
✅IdentitiesOnly yes is essential — without it SSH may try other loaded keys and silently authenticate as the wrong account.
Create and open the file
# Create the file if it doesn't exist
touch ~/.ssh/config

# Set required permissions (SSH will refuse to work otherwise)
chmod 600 ~/.ssh/config

# Open with nano or VS Code
nano ~/.ssh/config
code ~/.ssh/config
4

Test both SSH connections

Always do this before cloning or pushing. If either test fails, fix it here before moving on.

# Test personal account
ssh -T git@github-personal
▶ Hi your-personal-username! You've successfully authenticated...

# Test work account
ssh -T git@github-work
▶ Hi your-work-username! You've successfully authenticated...
⚠️Got Permission denied (publickey)? The public key was not added to that GitHub account. Re-check Step 2. Run ssh -vT git@github-personal for detailed debug output.
5

Clone a repository

Never use the default git@github.com:... URL. Always replace github.com with your alias. Then set your local identity immediately.

Clone a personal repo personal
git clone git@github-personal:your-personal-username/repo-name.git
cd repo-name

# Set identity immediately after cloning
git config user.name  "Your Name"
git config user.email "personal@email.com"
Clone a work repo work
git clone git@github-work:your-work-username/repo-name.git
cd repo-name

# Set identity immediately after cloning
git config user.name  "Your Name"
git config user.email "you@company.com"
💡Pattern to remember: Take any GitHub clone URL → replace github.com with github-personal or github-work. That's the only change needed.
6

Push a new local project to GitHub

First create an empty repository on GitHub.com (no README, no .gitignore). Then follow all 7 steps below.

terminal — new project personal
# 1 — Navigate to your project folder
cd your-project

# 2 — Initialise Git
git init

# 3 — Set local identity BEFORE the first commit
git config user.name  "Your Name"
git config user.email "personal@email.com"

# 4 — Stage all files
git add .

# 5 — First commit
git commit -m "Initial commit"

# 6 — Add remote using the SSH alias (NOT github.com)
git remote add origin git@github-personal:your-personal-username/repo-name.git

# 7 — Push
git branch -M main
git push -u origin main
terminal — new project work
# 1 — Navigate to your project folder

# 2 — Initialise Git

# 6 — Add remote using the SSH alias (NOT github.com)
git remote add origin git@github-work:your-work-username/repo-name.git

# 7 — Push
git branch -M main
git push -u origin main
7

Create and push a feature branch

The professional workflow — never commit directly to main. Create one branch per feature or bug fix, push it, then open a pull request.

terminal — feature branch workflow
# 1 — Start from an up-to-date main
git checkout main
git pull origin main

# 2 — Create and switch to a feature branch
git checkout -b feature/your-feature-name

# 3 — Write code, then review what changed
git status          # which files changed
git diff            # line-by-line diff

# 4 — Stage changes
git add .                            # all files
git add src/components/Button.jsx  # or a specific file

# 5 — Commit with a clear message
git commit -m "feat: add login button component"

# 6 — Push branch and open a Pull Request
git push -u origin feature/your-feature-name
# GitHub shows a banner → click "Compare & pull request"

# 7 — After PR is merged, clean up
git checkout main
git pull origin main
git branch -d feature/your-feature-name

8

Daily workflow checklist

Run this at the start of every session to avoid committing with the wrong identity.

  • 1
    Verify remote URL
    Run git remote -v — must show github-personal or github-work. Never bare github.com.
  • 2
    Verify identity
    Run git config user.email — must match the account the remote belongs to.
  • 3
    Pull before branching
    Always git pull origin main before creating a new branch to avoid conflicts.
  • 4
    Branch for every task
    git checkout -b feature/task-name — never code directly on main.
  • 5
    Commit often, message well
    Small focused commits are easy to review and revert. Use conventional prefixes from Step 7.
# Paste this at the start of every session
git remote -v
git config user.email
git checkout main
git pull origin main
git checkout -b feature/todays-task

Common mistakes & fixes

✗
Using github.com directly
git@github.com:username/repo.git — SSH picks a random key, commits go to the wrong account.

Fix: Always use the alias:
git@github-personal:your-personal-username/repo.git
✗
Forgot to set user.email
Commits show the wrong author. Check with git log --oneline -3.

Fix: git config user.email "correct@email.com" then amend the last commit:
git commit --amend --reset-author --no-edit
✗
Wrong remote URL on existing repo

Fix personal: git remote set-url origin git@github-personal:username/repo.git
Fix work: git remote set-url origin git@github-work:username/repo.git
⚠️
Keys disappear after reboot
SSH agent resets on restart. Add this to your ~/.zshrc or ~/.bashrc:
eval "$(ssh-agent -s)" && ssh-add ~/.ssh/id_ed25519_personal && ssh-add ~/.ssh/id_ed25519_work

Troubleshooting

Permission denied (publickey)
# Verbose debug — shows which key SSH is trying
ssh -vT git@github-personal

# List keys currently loaded in agent
ssh-add -l

# Re-add keys if agent is empty
ssh-add ~/.ssh/id_ed25519_personal
ssh-add ~/.ssh/id_ed25519_work
Inspect current repo config
git config --list --local   # all local settings
git remote -v               # remote URL
git config user.name
git config user.email
git log --oneline -5        # recent commits with authors
Fix wrong author on last commit
# Amend before pushing
git commit --amend --reset-author --no-edit

# Already pushed? Force push (use carefully on feature branches only)
git push --force-with-lease origin <branch-name>
Fix SSH file permissions
# Permissions must be exactly this or SSH won't work
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519_personal
chmod 600 ~/.ssh/id_ed25519_work
chmod 600 ~/.ssh/config
ls -la ~/.ssh/   # verify
